## 3. Структура репозитория

### Ветки

- `main` — стабильная версия проекта (демонстрационная)
- `dev` — интеграционная рабочая ветка команды
- `feature/*` — ветки для реализации отдельных задач (см. [примеры наименования feature-веток](#5-последовательность-работы-с-веткой))

### Папки

```
/server      — backend часть проекта
/client      — frontend часть проекта
/database    — схема БД, миграции, SQL-скрипты
/.github     — шаблоны PR, CODEOWNERS, issue templates
README.md    — описание проекта
```

## 5. Последовательность работы с веткой

### 1. Обновить `dev`

```bash
git checkout dev
git pull
```

### 2. Создать ветку задачи

```bash
git checkout -b feature/short-task-name
```

Примеры:

```
feature/ui-registration
feature/auth-api
feature/schema-v1
```

### 3. Выполнить работу и сделать коммит (см. [правила именования коммитов](#6-правила-именования-коммитов))

```bash
git add .
git commit -m "feat: реализован API авторизации"
```

### 4. Отправить ветку в удалённый репозиторий

При первом пуше новой ветки необходимо установить связь между
локальной и удалённой веткой:

```bash
git push -u origin feature/short-task-name
```

При последующих пушах можно использовать: 

```bash
git push -u origin feature/short-task-name
```

### 5. Создать Pull Request → в `dev` (см. [правила создания Pull Request](#7-правила-pull-request))

После одобрения — merge в `dev`.

После завершения этапа разработки:
`dev` → Pull Request → `main`

## 6. Правила именования коммитов

Используется стиль **Conventional Commits**.

### Формат:

```
type: краткое описание
```

### Типы:

- `feat` — новая функциональность
- `fix` — исправление ошибки
- `refactor` — переработка кода
- `docs` — документация
- `style` — форматирование
- `test` — тесты
- `chore` — технические изменения

### Примеры корректных сообщений:

```
feat: реализован API регистрации пользователя
fix: исправлена ошибка расчета суммы заказа
refactor: переработана структура контроллеров
docs: обновлен README
chore: добавлен PR template
```

### Недопустимые сообщения:

```
fix
update
123
test
```

## 7. Правила Pull Request

Pull Request создается только:

- из `feature/*` → в `dev`
- из `dev` → в `main`

Прямые коммиты в `dev` и `main` запрещены.

### Каждый PR должен содержать:

- Краткое описание реализованной функциональности
- указание на соответствующее требование из SRS
- Список внесенных изменений

### Пример структуры PR:

```
## Краткое описание
Реализован API авторизации.

## Требование из SRS
FR-3: Система должна обеспечивать аутентификацию пользователей.

## Изменения
- Добавлен AuthController
- Добавлена JWT авторизация
- Добавлены unit-тесты
```
